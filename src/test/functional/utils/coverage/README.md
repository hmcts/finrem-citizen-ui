# UI Coverage & Click Tracking

This directory contains utilities for tracking UI element interactions and generating coverage reports during Playwright test execution.

## Overview

The coverage system captures two types of metrics:

### 1. **Click Tracking (UI Coverage)**
Tracks which buttons, links, checkboxes, and form elements are clicked during test execution. Useful for understanding which parts of the UI are exercised by your test suite.

**Key Files:**
- `clickTracker.ts` - Core click tracking class
- `reportGenerator.ts` - Generates HTML reports from click data

**How it works:**
1. `ClickTracker` injects a click event listener into every page
2. Records selector, text, aria-label, URL, and timestamp for each interaction
3. Saves JSON files per test to `coverage/click-coverage/`
4. `CoverageReportGenerator` aggregates all click data into an HTML report

### 2. **Code Coverage (Nyc/Istanbul)**
Instruments your TypeScript source code and measures statement/branch/function coverage during test execution.

**Configuration File:**
- `.nycrc.json` - Nyc/Istanbul configuration (at project root)

**How it works:**
1. Nyc instruments all `src/**/*.ts` files (excluding tests)
2. Coverage data is collected during test runs
3. Reports are generated in `coverage/code-coverage/`

## Usage

### Generate Click Coverage Report After Tests

```bash
# Run functional tests and then generate click coverage report
yarn test:full-functional:coverage

# Or just generate a report from existing click data
yarn test:coverage-report
```

Report outputs to: `coverage/reports/ui-coverage.html`

### Generate Code Coverage with Nyc

```bash
# Example: Run tests with code coverage instrumentation
NODE_OPTIONS="--require ts-node/register --require @babel/register" \
  nyc \
  --reporter=html \
  --reporter=text \
  playwright test --config playwright.config.mts --project chromium --grep @PR
```

Or add a dedicated script to `package.json`:

```json
"test:coverage": "nyc playwright test --config playwright.config.mts"
```

## Test Integration

The `clickTracker` fixture is auto-initialized for all tests via `src/test/fixtures/fixtures.ts`:

```typescript
// In any test file:
import { test } from './fixtures/fixtures';

test('User clicks elements', async ({ clickTracker, page }) => {
  // Click tracking starts automatically
  await page.click('button');
  
  // Data is saved automatically after test completes
  // Access tracked clicks if needed:
  const clicks = await clickTracker.getClicks();
  console.log(clicks);
});
```

## CI/CD Integration

### Example: Add to Jenkins/GitHub Actions

```yaml
- name: Run Functional Tests with Coverage
  run: yarn test:full-functional:coverage

- name: Upload Coverage Artifacts
  run: |
    tar -czf coverage.tar.gz coverage/
    # Upload to artifact storage or S3
```

### Track Thresholds

Update `check-coverage` and threshold values in `.nycrc.json`:

```json
{
  "check-coverage": true,
  "lines": 70,
  "functions": 70,
  "branches": 60,
  "statements": 70
}
```

With `check-coverage: true`, the test suite will **fail** if coverage drops below specified thresholds.

## Analyzing Reports

### Click Coverage HTML Report
- **Coverage Summary Table** - Shows all unique UI elements clicked, how many times each was clicked
- **Test Details** - Expands each test to show the exact sequence of clicks
- **Statistics** - Total tests, clicks, and unique elements

### Code Coverage Reports
- Generated by Nyc in `coverage/code-coverage/`
- `index.html` - Interactive HTML report with file-by-file breakdown
- `coverage.json` - Machine-readable coverage data for CI integrations
- `lcov.info` - Standard format for coverage tools

## Troubleshooting

**Q: No click data in report?**
- Ensure tests use the `clickTracker` fixture
- Verify `coverage/click-coverage/` directory exists with JSON files
- Check browser console for JavaScript errors during test

**Q: Code coverage not working?**
- Ensure `.nycrc.json` exists at project root
- Verify `src/` directory structure matches exclude patterns
- Check that test files are properly excluded

**Q: Large coverage reports on CI?**
- Click data files can grow large with many tests
- Consider archiving `coverage/` between runs or uploading to external storage
- Generate report on-demand rather than storing raw data
